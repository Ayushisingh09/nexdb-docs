Architecture

NexDB is an embedded document database for Rust — an in-memory document store backed by a persistent write-ahead log, with optional SQLite durability, indexing, a built-in TCP server, and a migration engine.

Storage Engine

The storage engine manages a set of named collections, each containing a set of documents indexed by _id. The entire store lives in memory for maximum performance.

type Store = HashMap<String, HashMap<String, Value>>;
//           collection ──┘         └── document ID

Lookup by collection name and document ID is O(1) average case. Persistence is provided by the WAL — every mutation is written to disk before being applied in memory.

WAL & Checkpoint

Write-Ahead Log

Each mutation generates a JSON line in the WAL file (dbname.nex.wal):

{"type":"Insert","id":"uuid","document":{...}}
{"type":"Update","id":"uuid","document":{...}}
{"type":"Delete","id":"uuid"}

On NexDb::open(), the WAL is replayed in order to reconstruct any state that was not yet persisted. This guarantees crash safety.

Checkpointing

Over time the WAL grows. The checkpoint() method:

  1. Serializes the entire in-memory store as a single Snapshot entry
  2. Truncates the existing WAL file
  3. Future mutations start fresh

This bounds WAL size while preserving crash-safety. Safe to call at any time — does not block reads.

Concurrency Model

The NexDb instance is guarded by a std::sync::Mutex. All public API methods acquire the lock internally:

pub struct NexDb {
    inner: Mutex<NexDbInner>,
    ...
}

Each operation holds the lock for its duration. Operations are fast (in-memory + WAL append) so contention is low for most workloads. The TCP server spawns one thread per connection, each acquiring the lock per request — allowing concurrent reads while serializing writes.

Indexing

NexDB uses in-memory B-tree indexes (BTreeMap-based) for efficient querying:

  • Field paths: Indexes on simple fields ("age") and nested paths ("address.city")
  • Automatic maintenance: Updated transparently on insert, update, delete
  • Query optimization: Queries matching an indexed field use the index first, then apply remaining filters

Without index: O(n) full scan. With index: O(log n) B-tree lookup.

Query Engine

The QueryBuilder processes documents through a filter pipeline, sorts matches, applies skip/limit, and returns the result set.

Multiple fields in a filter are combined with AND logic.

TCP Server

The TCP server runs on a dedicated thread with std::net::TcpListener. Each accepted connection is handled in its own thread:

listener.incoming().for_each(|stream| {
    let db = db.clone();
    thread::spawn(move || handle_connection(db, stream));
});

Each line of input is parsed as JSON, routed by method to a handler, and a JSON response is written back. With --require-auth, every request is JWT-validated before processing.

Methods supported: insert, get, update, delete, query, all, count.

Migration Engine

The migration engine provides full data portability:

  • Dump/Restore: Serializes all collections to a portable JSON interchange file with metadata (collection names, document counts).
  • Copy: Reads all data from source, writes to target — useful for merging or splitting.
  • SQL Export: Generates CREATE TABLE and INSERT statements.
  • Auto-Import: Detects JSON array, CSV (with headers), or dump format — imports accordingly.
