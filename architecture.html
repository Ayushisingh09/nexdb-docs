Architecture

System Overview

NexDb is an in-memory-first, embeddable, document-oriented database written entirely in safe Rust. It combines a lock-free (no GC) memory model with a Write-Ahead Log for durability.

Storage Engine

The storage engine is organized as a two-level map:

NexDb {
    collections: RwLock<HashMap<String, Arc<RwLock<Collection>>>>,
    wal:         Mutex<WalWriter>,
    path:        PathBuf,
}

Collection {
    name: String,
    docs: BTreeMap<String, Document>,
    index_manager: IndexManager,
}

Key Design Decisions

  • BTreeMap over HashMap: Ordered iteration enables range scans and prefix queries. The O(log n) insert/lookup cost is negligible for in-memory workloads.
  • Two-level locking: The outer RwLock guards the collection registry (write-locked only for DDL). The per-collection RwLock allows concurrent reads on the same collection while serializing writes.
  • Arc sharing: Cloning the Arc lets operations hold a reference to a collection without keeping the registry lock, preventing lock convoy on the central map.

Write-Ahead Log (WAL)

Every mutation passes through the WAL before modifying in-memory state.

WAL Format

The WAL is an append-only JSON-lines file. Each line is a self-describing entry:

{"ts":1719571200,"c":"users","op":"insert","id":"u1","doc":{"name":"Alice","age":30}}
{"ts":1719571201,"c":"users","op":"update","id":"u1","doc":{"name":"Alice","age":31}}
{"ts":1719571202,"c":"users","op":"delete","id":"u1","doc":null}

Write Path

  1. Acquire collection write lock
  2. Serialize the WalEntry (with timestamp, operation, document)
  3. Acquire WAL mutex
  4. write_all() + sync_all() (fsync) to disk
  5. Release WAL mutex
  6. Apply mutation to in-memory BTreeMap
  7. Update indexes
  8. Release collection write lock

Recovery Path

On NexDb::open():

  1. Create empty in-memory state
  2. Open WAL file
  3. Read entries line-by-line
  4. For each entry: create collection if missing, apply insert/update/delete
  5. Result: in-memory state == last committed state

Concurrency Model

Lock Hierarchy

Strict ordering prevents deadlocks:

LevelLockProtectsAcquired By
1RwLock (tokio)Collection registry HashMapcreate_collection, drop_collection
2RwLock (tokio)Per-collection BTreeMapAll CRUD operations
3Mutex (tokio)WAL file appendAll mutations

Data Race Prevention

  • Zero unsafe: The entire codebase uses safe Rust. The compiler enforces Send + Sync at compile time.
  • Ownership discipline: Collection.docs is only accessible through &self (read lock) or &mut self (write lock) guards obtained from the RwLock.
  • Arc + Clone: Documents returned to callers are cloned while holding the lock, then the lock is dropped вҖ” no borrowed references escape.

Contention Under Load

  • Read-heavy: Unlimited concurrent readers per collection.
  • Write-heavy: Serialized per collection + funneled through global WAL mutex.
  • Cross-collection: Fully parallel вҖ” different collections have independent locks.

Indexing Layer

Field indexes are implemented as inverted indexes over a BTreeMap:

FieldIndex {
    name: String,
    field_path: String,       // e.g., "address.city"
    entries: BTreeMap<String, BTreeSet<String>>,
    //         field_value_json в”Җв”Җ> set of doc_ids
}

Key operations:

  • Insert: Extract field value via dot-path, serialize to JSON string, add doc ID to set.
  • Remove: Reverse of insert.
  • find_eq: Direct BTreeMap lookup by exact serialized value.
  • find_range: BTreeMap::range(low..=high) вҖ” lexicographic ordering of JSON strings.
  • Rebuild: Clear and re-insert all documents (used after WAL replay).

Threading Model

NexDb uses tokio's multi-threaded runtime for async I/O and concurrent request handling.

  • Async primitives: tokio::sync::RwLock and tokio::sync::Mutex are designed to be held across .await points.
  • I/O offloading: WAL fsync operations are dispatched to tokio's blocking thread pool, never blocking the async reactor.
  • Work stealing: Tokio's work-stealing scheduler automatically balances CRUD operations across available CPU cores.
  • No unsafe: All thread-safety is guaranteed by Rust's type system, not by manual reasoning.

File Format (.nexdb)

The .nexdb file is a plain JSON-lines log. Each line is a complete WAL entry:

FieldTypeDescription
tsi64Unix epoch timestamp (seconds)
cstringCollection name
op"insert" | "update" | "delete"Operation type
idstringDocument ID
docobject | nullDocument body (null for deletes)

The file is append-only. On recovery, all entries are replayed in order. Future versions will implement WAL checkpointing to prevent unbounded growth.
