Architecture

Overview

NexDB is an embedded document database for Rust. At its core, it is an in-memory document store backed by a persistent write-ahead log (WAL) and optional SQLite durability.

Storage Engine

The storage engine is the heart of NexDB. It manages a collection of named collections, each of which contains a set of documents, both indexed by their _id field.

In-Memory Store

Documents are stored in a nested HashMap structure:

type Store = HashMap<String, HashMap<String, Value>>;
//           collection name ──┘              │
//                     document ID ────────────┘

This provides O(1) average-case lookup by collection and document ID. The entire store is kept in memory for maximum read performance.

Persistence

Persistence is achieved through the WAL. Every mutation is appended to the WAL file before being applied in memory. On restart, the WAL is replayed to reconstruct the in-memory state.

WAL & Checkpoint

Write-Ahead Log

Each mutation generates a JSON line in the WAL file (dbname.nex.wal). The format is:

{"type":"Insert","id":"uuid","document":{...}}
{"type":"Update","id":"uuid","document":{...}}
{"type":"Delete","id":"uuid"}

On NexDb::open(), all WAL entries are replayed in order. This guarantees that no committed write is lost, even after a crash.

Checkpointing

Over time, the WAL grows as more mutations are applied. The checkpoint() method:

  1. Serializes the entire in-memory store to a new WAL file as a single Snapshot entry
  2. Truncates the existing WAL
  3. Future mutations are appended to the fresh WAL

This keeps WAL file size bounded while preserving full crash-safety. Checkpointing is safe to call at any time and does not block reads.

Concurrency Model

The NexDb instance is guarded by a Mutex from std::sync. All public API methods acquire the lock internally:

pub struct NexDb {
    inner: Mutex<NexDbInner>,
    ...
}

This design was chosen for simplicity and safety. Each operation holds the lock for the duration of the operation (read or write). In practice, operations are fast (in-memory + WAL append), so contention is low for most workloads.

For the TCP server, each connection runs on its own thread and acquires the lock per request. This allows concurrent reads while serializing writes through the mutex.

Indexing

NexDB supports B-tree indexes on document fields for efficient querying:

  • B-tree structure: indexes are implemented as in-memory B-trees using the btreemap module
  • Field paths: indexes can be created on simple fields ("age") or nested paths ("address.city")
  • Automatic maintenance: indexes are updated transparently on insert, update, and delete
  • Query optimization: when a query filter includes an indexed field, the index is used for the lookup and then results are filtered further

Without an applicable index, queries perform a full collection scan (O(n)). With an index on the filtered field, lookups are O(log n) via the B-tree.

Query Execution

The query engine processes each document through the filter pipeline, sorts the matching results, applies skip/limit, and returns the result set. When a matching index is found, documents are first narrowed via the index before the full filter is applied.

TCP Server

The TCP server runs on a dedicated thread and accepts connections using standard std::net::TcpListener. Each connection is handled in its own spawned thread:

listener.incoming().for_each(|stream| {
    let db = db.clone();
    thread::spawn(move || handle_connection(db, stream));
});

Each line of input is parsed as JSON, routed by the method field to the appropriate handler, and a JSON response is written back. If authentication is enabled via --require-auth, every request is validated against the JWT secret before processing.

Migration Engine

The migration engine provides data portability:

  • Dump/Restore: serializes all collections to a portable JSON file. The dump format includes metadata (collection names, document counts) along with the documents themselves.
  • Copy: reads all data from one database and writes it to another, useful for merging or splitting databases.
  • SQL Export: generates SQL CREATE TABLE and INSERT statements representing the current database state.
  • Auto-Import: detects the input format (JSON array, CSV with headers, or a dump file) and imports accordingly.

File Format

NexDB uses two files on disk:

FilePurposeFormat
mydb.nexSQLite database file (optional durability layer)SQLite format
mydb.nex.walWrite-ahead log for crash safetyJSON lines, UTF-8

The WAL file is the primary durability mechanism. When checkpointing, the WAL is compacted into a snapshot entry and truncated.
