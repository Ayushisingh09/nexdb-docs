Getting Started

Installation

Add the core library to your Cargo.toml:

[dependencies]
nexdb-core = "0.1.0"
serde_json = "1.0"

Enable optional BSON support:

nexdb-core = { version = "0.1.0", features = ["bson"] }

Install the CLI tool via cargo:

cargo install nexdb-cli

Or run directly with cargo:

cargo nexdb --help

Quick Start

Open or create a database, insert a document, and query it — all in a few lines:

use nexdb_core::NexDb;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let db = NexDb::open("mydb.nex")?;

    let collection = db.collection("users");
    let id = collection.insert(serde_json::json!({
        "name": "Alice",
        "age": 30,
        "city": "Berlin"
    }))?;

    let user = collection.get(&id)?;
    println!("Found: {}", user);
    Ok(())
}

Using the CLI:

cargo nexdb insert mydb.nex users '{"name":"Alice","age":30,"city":"Berlin"}'
cargo nexdb get mydb.nex users <id>

CRUD Operations

Insert

collection.insert(json!({"title": "Hello", "views": 42}))?;

Get by ID

let doc = collection.get(&doc_id)?;

Update

collection.update(&doc_id, json!({"views": 43}))?;

Delete

collection.delete(&doc_id)?;

List all

for doc in collection.all()? {
    println!("{:?}", doc);
}

Query Builder

The fluent QueryBuilder lets you compose rich queries with filters, sorting, and pagination:

use nexdb_core::query::QueryBuilder;

let results: Vec<serde_json::Value> = collection
    .query()
    .where_clause(r#"{"age": {"$gte": 21, "$lt": 65}, "city": "Berlin"}"#)?
    .sort("name", true)   // ascending
    .skip(0)
    .limit(20)
    .run()?;

The where_clause method accepts a JSON string. Supported operators: $eq, $ne, $gt, $gte, $lt, $lte, $in, $nin, $exists, $regex.

You can also parse directly from a serde_json::Value:

let filter: serde_json::Value = serde_json::from_str(r#"{"age": {"$gte": 21}}"#)?;
let results = collection.query().where_value(&filter)?.run()?;

Indexes

B-tree indexes accelerate queries on specific document fields:

// Create an index
collection.create_index("age")?;
collection.create_index("city")?;

// Query uses the index automatically
let results = collection.query()
    .where_clause(r#"{"age": {"$gte": 21}}"#)?
    .run()?;

// List and drop indexes
let indexes = collection.list_indexes()?;
collection.drop_index("age")?;

WAL & Checkpoint

The write-ahead log records every mutation before it is applied. This guarantees crash-safety:

// WAL is enabled by default — every write is logged before execution
// Manually checkpoint (snapshot + truncate WAL)
db.checkpoint()?;

The WAL log is stored at mydb.nex.wal alongside the database file. During recovery, any pending WAL entries are replayed automatically.

TCP Server

Start a TCP server that accepts JSON-line requests:

cargo nexdb serve --db mydb.nex --port 9480 --host 127.0.0.1

Each incoming line is a JSON object with method and params:

{"method":"insert","params":{"collection":"users","document":{"name":"Charlie","age":35}}}
{"method":"get","params":{"collection":"users","id":"a1b2c3..."}}
{"method":"query","params":{"collection":"users","filter":{"age":{"$gte":21}}}}
{"method":"delete","params":{"collection":"users","id":"a1b2c3..."}}

Responses are JSON-line objects:

{"ok":true,"id":"a1b2c3..."}
{"ok":true,"document":{...}}
{"ok":true,"documents":[...]}
{"ok":true}

Enable authentication with --require-auth:

cargo nexdb serve --db mydb.nex --require-auth --jwt-secret "mysecret"

Clients then include an auth field with their JWT token in each request.

Import / Export

Bulk operations for JSON and CSV formats:

# Import an array of documents from JSON
cargo nexdb import mydb.nex users data.json

# Export a collection to JSON
cargo nexdb export mydb.nex users data.json

# Import CSV (first row must be headers)
cargo nexdb import-csv mydb.nex users data.csv

# Export collection to CSV
cargo nexdb export-csv mydb.nex users data.csv

Programmatic usage:

let docs = db.collection("users").all()?;
let json = serde_json::to_value(&docs)?;
std::fs::write("users.json", serde_json::to_string_pretty(&json)?)?;

Migration Engine

The migrate command family helps move data between databases and formats:

# Dump a database to a portable JSON interchange file
cargo nexdb migrate dump mydb.nex dump.json

# Restore from a dump into a new database
cargo nexdb migrate restore newdb.nex dump.json

# Copy directly between two databases
cargo nexdb migrate copy source.nex target.nex

# Generate SQL schema from the database structure
cargo nexdb migrate to-sql mydb.nex schema.sql

# Auto-import: detect format and import
cargo nexdb migrate auto-import mydb.nex data.json

Use programmatically:

use nexdb_core::migrate::MigrationEngine;

let engine = MigrationEngine::new(&db);
engine.dump_to_file("backup.json")?;
engine.restore_from_file("backup.json")?;

Shell Completions

Generate shell auto-completion scripts:

# Bash
cargo nexdb completions bash > /etc/bash_completion.d/nexdb

# Zsh
cargo nexdb completions zsh > /usr/local/share/zsh/site-functions/_nexdb

# Fish
cargo nexdb completions fish > ~/.config/fish/completions/nexdb.fish

# PowerShell
cargo nexdb completions powershell > profile.ps1

# Elvish
cargo nexdb completions elvish > ~/.elvish/completions.eld

Clean Command

Remove database files when they are no longer needed:

# Remove a specific database and its WAL
cargo nexdb clean mydb.nex

# Remove all NexDB databases in a directory
cargo nexdb clean --all ./data/

BSON Support

When the bson feature is enabled, NexDB can decode BSON documents into its internal JSON-based representation:

[dependencies]
nexdb-core = { version = "0.1.0", features = ["bson"] }

BSON documents inserted via the TCP server or API are automatically decoded. The feature gate keeps the default dependency footprint minimal.

Benchmarks

NexDB includes a benchmark suite built on criterion:

# Run all benchmarks
cargo bench

# Run specific benchmark
cargo bench --bench crud
cargo bench --bench wal

Benchmarks cover: insert throughput, read latency, update performance, delete performance, indexed queries, and WAL write throughput. Results are reported in operations/second and latency percentiles.
